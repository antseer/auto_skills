# DeFi 资金池真实净收益率拆解器

## 功能说明

当用户看到高收益 DeFi 资金池宣传或需要对比多个协议的真实收益率时，自动将表面 APY 拆解为基础交易费、代币激励、借贷利息等组成部分，并估算代币激励稀释、Gas 成本、无常损失风险等隐性成本，输出风险调整后的净收益率。每个核心结论附带多层因果链推理和明确的失效条件。替代用户每次 2-3 小时的手动拆解计算。

## 用户画像

### 画像 1: Carol — DeFi 收益策略研究员
- **角色**: 专注 DeFi 收益优化的独立研究员，为自己和小型投资者群提供策略报告
- **日常工作流**: 每天在 DefiLlama + Dune + CoinGecko 之间切换，手动拉取 TVL/APY/链上数据并在 Jupyter Notebook 中计算净收益率
- **工具栈**: DefiLlama API, Dune Analytics, Python/Jupyter, TradingView
- **痛点**: 每次拆解一个资金池的真实 APY 需要 2-3 小时手动操作，API 限流和数据格式不一致导致维护成本极高；上个月差点推荐一个 Arbitrum USDC-ETH 池（宣传 85% APY，实际净 APY 只有约 7%）

### 画像 2: 小戴 — 加密新手散户
- **角色**: 在 CashApp 上持有少量 BTC 的普通上班族，通过 Twitter/Reddit 获取信息
- **日常工作流**: 刷推特看到高 APY 的 DeFi/DePIN 项目宣传 → 心动 → 不知道如何验证 → 要么冲动买入要么错过
- **工具栈**: CashApp, Twitter, Reddit, YouTube
- **痛点**: 上个月因看到某 Solana DePIN 项目"月收入 $500K"而买入代币，后来发现 80% 收入是代币激励而非真实付费；完全缺乏分辨"面值"和"实值"的能力

## 用户故事

1. **As Carol**, I want to input a DeFi pool address and get an automated APY decomposition with causal chain analysis showing *why* each component is at its current level, so that I can produce accurate strategy reports in 15 minutes instead of 2-3 hours.
2. **As 小戴**, I want to know whether a DePIN/DeFi project's advertised high yield is real or mostly token incentives, so that I don't lose money on projects that look profitable on the surface but are actually subsidized by inflationary tokenomics.

## 适用场景

- 看到 DeFi 协议或资金池的高 APY 宣传时
- 对比多个 DeFi 协议的收益率选择最优策略时
- DePIN 项目声称高收入但需验证"真实用户付费 vs 代币激励"比例时
- 定期评估已投入的 DeFi 头寸的实际回报时
- 监管变化（如 CLARITY Act）后评估 CeFi-DeFi 收益率利差变化时

## 输入参数

| 参数 | 必填 | 说明 | 默认值 |
|------|------|------|--------|
| 协议/资金池 | 是 | DeFi 协议名称或资金池描述 | — |

| period | 否 | 评估周期 | 90 天 |

## 执行步骤

### Step 1: 识别目标协议/代币并获取基本信息
调用 `ant_market_sentiment(query_type='topics_list')` 搜索用户指定的 DeFi 协议/代币相关话题。
对匹配话题调用 `ant_market_sentiment(query_type='topic_detail', topic_id=<id>)` 获取协议基本信息和市场情绪。
调用 `ant_market_sentiment(query_type='coins_list')` 搜索协议治理代币。
对匹配代币调用 `ant_market_sentiment(query_type='coin_detail', coin_id=<id>)` 获取代币详情。

### Step 2: 获取代币价格趋势（用于评估代币激励的真实价值）
调用 `ant_market_sentiment(query_type='coin_time_series', coin_id=<治理代币id>)` 获取治理代币价格历史。
计算代币在用户指定周期内（默认 90 天）的价格变化率。
如果代币价格下跌趋势明显，标注"代币激励部分 APY 存在稀释风险"。

### Step 3: 获取链上资金流向数据
如果协议代币在支持列表中：
调用 `ant_fund_flow(query_type='exchange_netflow', coin=<代币>)` 获取代币交易所净流向。
调用 `ant_fund_flow(query_type='exchange_reserve', coin=<代币>)` 获取交易所储备变化。

**因果链分析（必须执行）**: 不要仅报告"大量代币流入交易所"这一观测事实。必须构建至少 2 层因果推理：
- **第 1 层（现象→原因）**: 代币大量流入交易所 → 为什么？→ 激励代币正在被领取者卖出套现（而非长期持有），说明激励吸引的是"挖提卖"用户而非协议的真实使用者
- **第 2 层（原因→后果）**: 持续卖压 → 代币价格下行趋势 → 以代币计价的激励 APY 的 USD 实际价值持续缩水 → 当前宣传的激励 APY 可能在评估周期内大幅衰减
- **第 3 层（后果→反转条件）**: 如果代币流向反转（净流出交易所）或协议推出代币锁仓/veToken 机制，卖压可能减缓 → 此时激励 APY 的衰减预期需要上调

### Step 4: 获取市场波动率数据（用于估算无常损失风险）
对资金池涉及的主要代币调用:
- `ant_market_indicators(query_type='atr', coin=<coin>)` 获取 ATR 作为波动率代理
- `ant_market_indicators(query_type='boll', coin=<coin>)` 获取布林带宽度作为波动率辅助指标
用历史波动率估算 AMM 池的无常损失期望值:
- 低波动率（ATR < 均值）: 无常损失估计 1-3%
- 中波动率（ATR 接近均值）: 无常损失估计 3-8%
- 高波动率（ATR > 1.5倍均值）: 无常损失估计 8-20%

### Step 4b: 获取协议 TVL 和收益数据（跨源交叉验证）
调用 `ant_protocol_tvl_yields_revenue(query_type='protocol_tvl', protocol=<协议名>)` 获取协议 TVL 数据。
调用 `ant_protocol_tvl_yields_revenue(query_type='protocol_yields', protocol=<协议名>)` 获取协议收益率数据。
调用 `ant_protocol_tvl_yields_revenue(query_type='protocol_revenue', protocol=<协议名>)` 获取协议收入数据。

**跨源交叉验证规则（强制执行）**:
- 将 Step 4b 的协议收益率数据与 Step 2 的代币价格趋势进行交叉验证：如果协议层面的收益率在下降但宣传 APY 不变，说明 APY 的维持依赖于增加代币激励发放量 → 不可持续信号
- 将 Step 3 的资金流向数据与 Step 4b 的 TVL 变化趋势交叉验证：如果 TVL 在下降但代币流入交易所增加，说明用户在撤出流动性并抛售激励代币 → 双重看空信号
- **置信度规则**: 只有当 ≥2 个独立 MCP 数据源（`ant_market_sentiment` 的代币趋势、`ant_fund_flow` 的资金流向、`ant_protocol_tvl_yields_revenue` 的 TVL/收益、`ant_market_indicators` 的波动率）指向同一方向时，才能输出"高置信度"结论。如果数据源之间存在矛盾，必须标注为"中/低置信度"并列出矛盾点

### Step 5: 构建 APY 拆解分析
基于收集的数据，构建以下拆解框架:
1. **宣传 APY**: 用户输入或从话题详情中提取
2. **基础交易费/借贷利息 APY**: 当 MCP 无法直接获取池级数据时，使用以下参考值估算：AMM 池基础交易费约为宣传 APY 的 10-20%（主流稳定币对偏低、波动对偏高）；借贷协议借贷利率 = 当前 Aave/Compound 同类资产利用率对应利率（约 2-8%）；DePIN 基础收入 = 询问用户提供或从话题中提取"真实付费用户收入"数字
3. **代币激励 APY**: = 宣传 APY - 基础 APY
4. **代币激励实际价值调整**: 代币激励 APY × (1 + 代币价格变化率)
5. **无常损失估计**: 基于 Step 4 波动率分析
6. **净 APY 估计**: 调整后基础 APY + 调整后激励 APY - 无常损失 - Gas 成本估算

**因果链总结（必须执行）**: 为净 APY 结论构建完整的因果推理链：
- 宣传 APY 高 → 为什么？→ 代币激励占比 [X]%（第 1 层）
- 代币激励占比高 → 后果是什么？→ 净收益严重依赖代币价格走势，而代币 [正在/没有] 经历卖压（第 2 层）
- 代币卖压存在 → 进一步后果？→ 持有期内激励的 USD 价值预期下降 [Y]%，净 APY 从宣传的 [X]% 降至约 [Z]%（第 3 层）

### Step 5b: 生成失效条件（每个核心结论必须附带）
对以下每个核心结论生成至少 1 个明确的失效条件：
1. **净 APY 估计的失效条件**: "此净 APY 估计失效，如果：(a) 协议在评估周期内改变代币激励发放率（历史上每 [X] 个月调整一次），(b) 治理代币价格在评估周期内反转上涨超过 [Y]%，(c) 协议所在链的 Gas 费出现结构性变化"
2. **无常损失估计的失效条件**: "此 IL 估计失效，如果：(a) 资金池涉及的代币出现超出历史波动率 2 倍以上的异常波动（黑天鹅事件），(b) 协议更换 AMM 曲线类型（如从 x*y=k 切换到集中流动性）"
3. **代币激励可持续性判断的失效条件**: "此判断失效，如果：(a) 协议宣布新一轮融资或代币回购计划改变供需结构，(b) 协议通过治理投票大幅削减激励预算，(c) 竞品协议的激励战争导致行业整体激励水平剧变"

### Step 6: 获取同类对比数据（结构化对比方法论）
调用 `ant_market_sentiment(query_type='category_topics', category_id=<defi相关category>)` 获取同赛道协议话题。
调用 `ant_protocol_tvl_yields_revenue(query_type='protocol_tvl', protocol=<同类协议名>)` 获取对比协议的 TVL 数据。
调用 `ant_protocol_tvl_yields_revenue(query_type='protocol_yields', protocol=<同类协议名>)` 获取对比协议的收益率数据。

**结构化对比方法论（必须执行）**:
对 2-3 个同赛道协议执行以下标准化对比：
1. **APY 拆解对比**: 每个对比协议也拆解为基础 APY + 激励 APY，确保同口径对比
2. **风险调整排名**: 按风险调整后净 APY 从高到低排序，并标注排名依据
3. **激励可持续性对比**: 对比各协议的代币激励占比和代币价格趋势，评估哪个协议的 APY 更可持续
4. **相对优势总结**: 明确回答"在 [用户评估周期] 内，哪个协议的风险调整后净收益最优，为什么"

**输入:**
```
这个 Arbitrum 上的 USDC-ETH 池子宣传 85% APY，是真的吗？
```

**输出:** 生成 APY 拆解报告，显示：
- **APY 拆解**: 宣传 85% APY 中基础交易费约 12%、代币激励约 73%（因治理代币 90 天跌 42%，调整后约 42%），扣除无常损失 -15%、Gas -3%，风险调整后净 APY 约 36%
- **因果链**: 高 APY 主要由代币激励驱动（第 1 层）→ 治理代币正经历持续卖压（交易所净流入增加 35%），导致激励的 USD 价值持续缩水（第 2 层）→ 如果协议推出 veToken 锁仓机制或代币回购，卖压可能缓解（第 3 层）
- **跨源验证**: 代币价格下跌（ant_market_sentiment）+ 交易所净流入增加（ant_fund_flow）+ TVL 增速放缓（ant_protocol_tvl_yields_revenue）三源一致指向看空，置信度: 高
- **失效条件**: 净 APY 估计失效如果治理代币价格在 90 天内反弹超过 50%，或协议将激励发放率削减超过 30%
- **同赛道对比**: 附 Aave/Compound 同赛道对比，按风险调整后净 APY 排名

## 反模式

1. **不要只看宣传 APY 做判断**: 任何不做拆解就推荐高 APY 池子的行为都是反模式。
2. **不要忽视代币价格趋势**: 代币激励占 APY 70%+ 时，代币价格趋势是净收益的最大变量。
3. **不要把无常损失当固定值**: IL 是概率估计，应给出范围而非单一数字。
4. **不要遗漏 Gas 成本**: 对小额头寸（<$1000），Gas 成本可能吞掉大部分收益。
5. **无常损失仅适用于 AMM 流动性池**: 纯借贷协议（如 Aave 单币存款）和 DePIN 协议无无常损失，Step 4 的 IL 估算对这类场景跳过，改为标注"协议集中度风险"。代币激励拆解（Step 2-3、Step 5）对所有场景均适用，包括 DePIN 的"真实用户付费 vs 代币补贴"分析。
6. **MCP 数据限制**: 当前 MCP 不提供池级 APY 组件数据，部分拆解需基于估算。
7. **不要输出没有失效条件的结论**: 每个核心结论（净 APY、IL 估计、激励可持续性）都必须附带至少 1 个明确的"此判断失效如果..."条件。省略失效条件会给用户虚假的确定感。
8. **不要在数据源矛盾时输出高置信度**: 当不同 MCP 数据源指向不同方向时，必须标注为中/低置信度并列出矛盾点。单一数据源得出的结论最高标注为"中置信度"。
